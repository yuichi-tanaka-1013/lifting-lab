# LLM Wiki — 個人ナレッジベースを LLM に「育てさせる」システム設計

## プレゼン情報

- **対象**: 社内エンジニア
- **所要時間**: 15-20分（想定）
- **ゴール**: 「RAG とは違う、LLM によるナレッジ蓄積パターン」の紹介と、実例によるデモ

---

## 1. 導入: RAG の限界（2分）

### 問題提起

- 既存の LLM + ドキュメント体験（ChatGPT ファイルアップロード、NotebookLM、RAG）は**毎回ゼロから検索・合成**している
- 5つのドキュメントを横断する質問 → 毎回同じ断片をかき集めて回答
- **知識が蓄積しない。構造化されない。複利が効かない。**

### この発表で紹介するもの

- LLM が **持続的な Wiki を自律的に構築・維持** するパターン
- 原案: Andrej Karpathy「LLM Wiki」パターン
- 実例: 自分の筋トレ復帰プロジェクトで 2 週間運用した結果

---

## 2. アーキテクチャ: 3 層構造（3分）

```
┌─────────────────────────────────────────┐
│  Layer 3: CLAUDE.md（スキーマ / 指示書）     │  ← 人間が書く。LLM の行動規範
├─────────────────────────────────────────┤
│  Layer 2: wiki/（LLM が完全管理）           │  ← LLM が書く。人間は読むだけ
│  ├── entities/    （人物・サプリ・種目）      │
│  ├── concepts/    （理論・メカニズム）        │
│  ├── protocols/   （プログラム・手順）        │
│  ├── sources/     （ソースサマリ）           │
│  ├── overviews/   （テーマ横断）            │
│  ├── personal/    （個人データ分析）         │
│  ├── references/  （動画リファレンス等）      │
│  ├── index.md     （カタログ）              │
│  └── log.md       （活動ログ）              │
├─────────────────────────────────────────┤
│  Layer 1: raw/（不変ソース）                │  ← 人間が投入。LLM は読むだけ
│  ├── articles/    （Web 記事）              │
│  ├── papers/      （論文）                  │
│  └── personal/    （InBody CSV、トレログ）   │
└─────────────────────────────────────────┘
```

### ポイント

- **Layer 1 は immutable**: LLM は読むだけ、書かない。Source of truth
- **Layer 2 は LLM が所有**: 人間は読むだけ。作成・更新・クロスリファレンスは全て LLM
- **Layer 3 が制御面**: LLM にドメイン知識、規約、オペレーション手順を伝える。人間と LLM で共進化する
- **全体は単なる Git リポジトリ**: バージョン管理、差分レビュー、コラボレーションが無料

## 3. オペレーション: 3 つの基本動作（3分）

### /ingest — 新しいソースを Wiki に統合

```
入力: Web 記事 URL or PDF
  ↓
LLM がソースを読む
  ↓
entity ページ更新（人物、種目、サプリ...）
concept ページ更新（理論、メカニズム...）
sources/ にサマリページ作成
矛盾チェック（既存の知見と衝突する場合は両論併記）
index.md, log.md 更新
  ↓
1 つのソースが 10-15 ページに波及
```

**デモ例**: Stronger By Science の Detraining 記事 → Detraining, Muscle Memory, Volume Landmarks, Greg Nuckols（新規）等の 8 ページに反映

### /query — Wiki に質問

```
質問: 「復帰最初の 4 週間で何をすべきか？」
  ↓
LLM が index.md → 関連ページを読む
  ↓
回答を合成（[[wikilink]] 付き）
  ↓
重要な回答は wiki/overviews/ に保存 → 知識として蓄積
```

### /lint — 健康診断

- オーファンページ（リンクされていない）
- 矛盾する記述
- 古い情報
- 埋められそうな知識の穴

## 4. 実例: 筋トレ復帰 Wiki（5分）

### 2 週間で構築されたもの

| 指標 | 値 |
|------|-----|
| Markdown ファイル数 | 61 |
| Git コミット数 | 10 |
| wiki ページ数 | entities 11, concepts 14, protocols 7, sources 1, overviews 2, personal 6, references 1 |
| 活動ログ | 121 行 |

### 具体的に何が起きたか（ストーリーで見せる）

1. **ベースラインヒアリング** → 1 問ずつ質問 → `raw/personal/baseline-2026-05.md` + `wiki/personal/current-state.md`
2. **外部ソース ingest** → SBS Detraining 記事 → 8 ページ更新、1 entity 新規作成
3. **/query でプログラム設計** → `wiki/overviews/return-program-week1-4.md`（Transition Gate 付き）
4. **実行結果のフィードバック** → Plan vs Actual の記録 → 計画超過のパターン検出 → アンカリングバイアスの指摘
5. **InBody CSV 投入** → 体組成の推定値を実測値で全面修正 → 戦略のシフト（筋量回復 → 体脂肪減少）
6. **食事記録** → カロリー・タンパク質を概算 → 「プロテイン 1 スクープ追加するだけ」という具体的アクション

### ここが面白い

- **知識が複利で増える**: ingest した記事の知見が、後の query で自動的に参照される
- **個人データと一般知識の統合**: InBody の実測値 + Detraining の論文 → 「あなたの場合は」という判断
- **矛盾の検出**: 計画 vs 実際の乖離、推定値 vs 実測値のギャップを wiki が記録し続ける
- **仮説の追跡**: 「なぜ ROM が保たれたか」→ 仮説ページ → 検証指標 → 自動フォローアップ

## 5. 技術スタック（2分）

| レイヤー | ツール | 役割 |
|---------|-------|------|
| LLM | Claude Code（Claude Opus 4.6） | Wiki の読み書き、分析、対話 |
| エディタ | Obsidian | Wiki の閲覧（グラフビュー、wikilink）。LLM が書いたものを人間が読む IDE |
| バージョン管理 | Git + GitHub | 変更履歴、diff レビュー |
| データ形式 | Markdown + YAML frontmatter | LLM が扱いやすく、人間も読みやすい |
| 検索 | index.md（小規模）/ grep | 100 ページ程度なら index.md で十分 |

### なぜ RAG を使わないか

- 100 ページ規模なら **index.md + grep で十分**。embedding インフラ不要
- wiki 自体が「事前に合成された知識」なので、生ドキュメントから毎回検索する必要がない
- 必要になったら qmd 等のローカル検索を後付けできる

## 6. このパターンの応用先（2分）

| 領域 | 例 |
|------|-----|
| **個人の学習** | 今回の筋トレ、資格勉強、読書ノート |
| **技術調査** | 新技術の評価、アーキテクチャ検討（論文 + ブログ + 社内知見を統合） |
| **チーム運用** | Slack スレッド + 議事録 + 設計書 → LLM が内部 Wiki を自動更新 |
| **競合分析** | プレスリリース + IR 資料 + ニュース → 競合の動きを構造化 |
| **インシデント管理** | ポストモーテム + Runbook + アラート履歴 → 障害知見の蓄積 |

### 共通する条件

- 知識が**時間とともに蓄積**される領域
- **クロスリファレンス**に価値がある（A と B の関連に気づくことが重要）
- 人間が**メンテナンスを放棄しがち**な領域（LLM はメンテナンスを厭わない）

---

## 7. 制約と注意点（2分）

| 制約 | 対策 |
|------|------|
| LLM のコンテキストウィンドウ | 大規模 wiki では index.md だけで長くなる → カテゴリ別 index に分割 |
| ハルシネーション | raw/ を Source of truth として維持。wiki の記述は常にソースに帰着可能 |
| コスト | Claude Code の利用料。ただし RAG インフラの構築・運用コストと比較すれば安い |
| プライバシー | 個人データ（体組成、食事）を LLM に渡す。センシティブなドメインでは考慮が必要 |
| スケール | 数百ページまでは index.md で十分。それ以上は検索インフラが必要 |

---

## 8. まとめ（1分）

1. **RAG は毎回検索、LLM Wiki は事前に合成** — 知識が複利で増える
2. **3 層構造**（raw / wiki / schema）で責務を分離
3. **Git リポジトリとして管理** — 特別なインフラ不要
4. **人間はソースと質問を提供、LLM がメンテナンス** — 人間が放棄しがちな作業を LLM に委譲
5. 個人プロジェクトからチーム運用まで応用可能

> 「LLM は検索エンジンではなく、知識のメンテナー（管理者）として使う」
