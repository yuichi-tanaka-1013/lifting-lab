5-min Lightning Talk

LLM に Wiki を
「育てさせる」

RAG を超える、知識が複利で増えるパターン

原案: Andrej Karpathy「LLM Wiki」パターン

Problem

RAG は毎回ゼロから検索する

ChatGPT ファイルアップロード、NotebookLM、社内 RAG ...

5 つのドキュメントを横断する質問 → 毎回同じ断片をかき集めて回答

知識が蓄積しない。構造化されない。複利が効かない。

Solution

3 層アーキテクチャ

CLAUDE.md スキーマ / 指示書 ← 人間が書く
↕ LLM の行動を制御
wiki/ entities, concepts, protocols, sources, personal ... ← LLM が書く
↑ /ingest で統合
raw/ 論文, 記事, CSV, トレーニングログ ← 人間が投入

全体はただの Git リポジトリ + Markdown。特別なインフラ不要。

System Detail

CLAUDE.md が LLM の行動を制御する

CLAUDE.md（Schema as Code）
  • ドメイン知識（「出戻り組は結合組織に注意」）
  • ページの frontmatter 規約（type, tags, confidence）
  • エビデンス階層（メタ分析 > MASS > 論文 > ...）
  • オペレーション手順（/ingest, /query, /lint）
  • 禁止事項（「過去の重量を引き合いに出さない」）

→ LLM はこのファイルを読んで「どう振る舞うか」を決定

wiki/ の内部構造
wiki/
├── entities/ ── 人物 7, サプリ 5
├── concepts/ ── 理論 14
├── protocols/ ── 手順 7
├── sources/ ── ソースサマリ 1
├── overviews/ ── テーマ横断 2
├── personal/ ── 個人データ 6
├── index.md ── 全ページカタログ
└── log.md ── 時系列の活動ログ

→ index.md で検索。embedding DB 不要

Data Flow

/ingest の流れ — 1 記事が 8 ページに波及

raw/articles/sbs-detraining.md （Detraining 論文レビュー記事）
│
├─→ wiki/sources/sbs-detraining.md ソースサマリ（新規）
├─→ wiki/entities/greg-nuckols.md 著者ページ（新規作成）
├─→ wiki/entities/eric-helms.md 相互リンク追加（更新）
├─→ wiki/concepts/detraining.md 筋力低下タイムライン追加（更新）
├─→ wiki/concepts/muscle-memory.md 再獲得速度の推定追加（更新）
├─→ wiki/concepts/volume-landmarks.md Bickel 2011 の知見追加（更新）
├─→ wiki/index.md カタログ更新
└─→ wiki/log.md 活動記録追記

1 つのソースが 既存ページのクロスリファレンスを自動更新 する。
これが RAG にはない「知識の複利」の仕組み。

Operations

3 つの基本動作

/ingest — ソースを統合

Web 記事 1 本を投入
→ LLM が読んで 10-15 ページを自動更新
→ クロスリファレンス、矛盾チェック

/query — Wiki に質問

質問 → index.md から関連ページを探索 → 回答合成
→ 良い回答は Wiki に保存 → 知識が複利で増える

/lint — 健康診断

オーファンページ検出
矛盾する記述の発見
知識の穴の提案

RAG: 毎回検索して合成

LLM Wiki: 事前に合成済み。知識が残る

Case Study — 2 weeks

筋トレ復帰 Wiki で起きたこと

ベースラインヒアリング → 体組成・怪我歴・生活から current-state.md を自動生成
論文レビュー記事を /ingest → 8 ページが連鎖更新、新規 entity ページ自動作成
トレーニング結果を報告 → 3 回連続の計画超過を検出 → アンカリングバイアスを LLM が指摘
InBody CSV を投入 → 体脂肪率の推定 26.9% → 実測 20.7% → 戦略が自動シフト
食事 3 日分を入力 → 「プロテイン 1 杯追加するだけ」という結論

2 週間で 61 ファイル / 10 コミット。個人データと一般知識が統合された Wiki が育った

Why this works

なぜこのパターンが強いか

RAG LLM Wiki
知識の蓄積 しない 複利で増える
クロスリファレンス 毎回発見 事前に構築済み
個人データ × 一般知識 分離 統合して判断
メンテナンスコスト 人間がやる LLM がやる（≈ 0）
インフラ embedding DB 等 Git + Markdown
  • 応用: 技術調査、チーム Wiki、競合分析、インシデント管理 ...
  • 条件: 知識が蓄積する領域 × クロスリファレンスに価値がある × メンテを人間が放棄しがち
LLM は検索エンジンではなく
知識のメンテナーとして使う
