# LLM に Wiki を「育てさせる」— 5分LT

---

## 1. 問題（30秒）

RAG（ChatGPT ファイルアップロード、NotebookLM 等）は **毎回ゼロから検索** する。
知識が蓄積しない。構造化されない。**複利が効かない。**

---

## 2. 解法: LLM Wiki パターン（1分）

LLM に **持続的な Wiki を構築・維持させる**。

```
raw/（人間が投入、LLM は読むだけ）
  ↓ /ingest
wiki/（LLM が書く、人間は読むだけ）
  ↑ /query
CLAUDE.md（LLM への指示書）
```

- ソースを投入 → LLM が読んで 10-15 ページを自動更新
- 質問する → Wiki から回答合成 → 良い回答は Wiki に保存 → 知識が複利で増える
- 全体はただの **Git リポジトリ + Markdown**。特別なインフラ不要

---

## 3. 実例（2分）— 2週間で起きたこと

筋トレ復帰の個人 Wiki を Claude Code で構築。

**Day 1**: 体組成・怪我歴をヒアリング → ベースライン作成

**ソース投入**: 論文レビュー記事を /ingest → Detraining, Muscle Memory 等 8 ページ自動更新

**プログラム設計**: /query で復帰プログラムを生成 → 結合組織の配慮、段階的な重量増加ルール

**実行 → フィードバック**: トレーニング結果を報告 → 3回連続で計画超過 → **LLM がアンカリングバイアスを指摘**

**InBody CSV 投入**: 体脂肪率の推定値 26.9% → 実測 20.7% に修正 → **戦略が「筋量回復」から「体脂肪減少」に自動シフト**

**食事記録**: 3日分の食事を入力 → カロリー・タンパク質を概算 → **「プロテイン1杯追加するだけ」** という結論

→ 2週間で **61ファイル、10コミット**。個人データと一般知識が統合された Wiki が育った。

---

## 4. なぜこれが面白いか（1分）

| RAG | LLM Wiki |
|-----|----------|
| 毎回検索 | 事前に合成済み |
| ドキュメント単位 | クロスリファレンス済み |
| 知識が増えない | **複利で増える** |

- **個人データ × 一般知識**: InBody の CSV + 論文の知見 → 「あなたの場合は」
- **矛盾の検出**: 推定値 vs 実測値、計画 vs 実績のギャップを Wiki が記録
- **メンテナンスコスト ≈ 0**: 人間が放棄する作業（クロスリファレンス、更新、整合性チェック）を LLM がやる

---
